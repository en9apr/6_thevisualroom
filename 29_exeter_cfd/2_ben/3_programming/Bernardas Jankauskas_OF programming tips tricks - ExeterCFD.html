OpenFOAM programming tips, tricks and various discoveries

Here I will share different finds inside OF code that might be helpful in your code.

Resetting dimensions

Usually empirical formulas in the papers are rarely dimensionally balanced. If one would like to use these formulas in their code there is a way of implementing that. This can be done with help of resetting dimensions function in OpenFoam.

Example:

Let's say we have an empirical parameter dependency on the temperature in the system:

As it can be seen in terms of dimensions this equation is unbalanced (pKsp is dimensionless whereas T is in Celcius) and in order to use these values we need to trick the code. 

 Foam::volScalarField Foam::liquidSolution::pKsp (volScalarField T)
 {
     volScalarField Tnew = T;
     Tnew.dimensions().reset(dimensionSet(0,0,0,0,0,0,0));
     return ((-0.0265*(Tnew-273.15))+13.972); 
 }

There is an alternative, yet slightly more involved way of doing this. It would involve running over the mesh cell by cell by hand and using the .value() function. 

 Foam::tmp<Foam::volScalarField> Foam::liquidSolution::pKsp (volScalarField T)
 {
     
     // Create a temporary field
     tmp<Foam::volScalarField> tpKsp
     (
         new volScalarField
         (
             IOobject
             (
                 "pKsp",
                 mesh().time().timeName(),
                 mesh(),
                 IOobject::NO_READ,
                 IOobject:NO_WRITE
             ),
             mesh(),
             dimensionedScalar("zero", dimless, 0.0)
         )
     );
     
     // Make a reference
     volScalarField& pKsp = tpKsp();
     
     // Loop over all cells and calculate value of tpKsp
     forAll(T, celli)
     {
         const scalar Ti = T[celli];
         pKsp[celli] = -0.0265*(Ti-273.15))+13.972
     }
     
     return tpKsp;
 }

Using mesh() to look for specific objects

There are times that we need to access a specific field in the object database (or as it's named in the code: objectRegistry). Looking into ojectRegistry file we can see that there are multiple lookup functions. Solvers don't usually have this database object flying in the open as it is a background process. Luckily mesh object has access to all of the lookup functions. Not entirely sure why, either it is because it is instantiated as IOobject or the access is coming from the parent classes in the hierarchy. My top guess is that it is related to IOobject, which then means that the following functions should be accessible with all IOobjects in your code (Will double check that at some point and edit this part). 

 //- Return the list of names of the IOobjects 
 // (eg. can use this in Info statement)
 mesh.names();
 
 //- Return the list of names of the IOobjects of given type (eg. volScalarField)
 //  eg. can use this in info statement
 
 mesh.names<class Type>() const;
 
 //- Lookup and return the object of the given Type
 //  eg. again can use this in info statement to see what objects of the type exist
 
 mesh.lookupObject<class Type>();
 
 //- Lookup and return the object of the given Type and name
 //  Probably the most useful as we can use it to get access to specific object
 //  eg. sf = mesh.lookupObject<DimensionedField<scalar, volMesh> >("RR.CH4")
 
 mesh.lookupObject<class Type>(object Name);

Plotting dimensionedField in Paraview

Sometimes your solver might be calculating something, but the object does not have a type that ParaView is able to plot. A specific example would be reaction rate values in chemistry libraries, that are only calculated in the internalField. Simply making the solver to write those values into a file wouldn't work, because boundaryField is required in order to plot the field. The way to get the parameter is to create a volScalarField with a dummy zeroGradient boundary field and internalField values from reaction rate.

As an example I will take reactingFoam solver and a counterFlowFlame2D case. In this tutorial one of the reactants is CH4 and it's reactionRate object is named RR.CH4.

We first start with createFields.H and at the bottom of it add: 

 createFields.H
 
 // New field to store reactionRate values
 volScalarField RRCH4
 (
     IOobject
     (
         "RRCH4",
         runTime.timeName(),
         mesh,
         IOobject::NO_READ,
         IOobject::AUTO_WRITE
     ),
     mesh,
     dimensionedScalar("zero", dimMass/dimVolume/dimTime, 0.0),
     zeroGradientFvPatchScalarField::typeName  // Force zeroGradient boundary condition
 );
 
 // Reference to internalField; will add reactionRate to this field
 scalarField& sf = RRCH4.internalField();

Next in the solver we move to YEqn.H. During runTime this is the part where reaction rate values are calculated. At the bottom we will add the line to store the calculated reactionRate for CH4 

 YEqn.H
 // Use mesh to lookup object of specific type and name
 sf = mesh.lookupObject<DimensionedField<scalar, volMesh> >("RR.CH4")

And there you go. Now the solver will be saving the values in a format that ParaView can understand.
